## What Was Accomplished

Fixed local development serving so the wiki site renders properly in the browser instead of triggering file downloads or showing directory listings.

### Files Changed

- **`wiki.cabal`** — added `warp` and `wai-app-static` to `build-depends`; also added `-threaded -rtsopts -with-rtsopts=-N` GHC options (fixes crash on `watch`/`server` commands which require the threaded runtime)
- **`Main.hs`** — added `serve` subcommand; added imports for `Network.Wai.Application.Static`, `Network.Wai.Handler.Warp`, `WaiAppStatic.Types`

### New `serve` Command

```bash
cabal run wiki -- serve          # port 8000 (default)
cabal run wiki -- serve 9000     # custom port
```

Implemented in `serveLocally :: Int -> IO ()`. Intercepts the `"serve"` arg before Hakyll sees it (via `getArgs` check at the top of `main`). Uses `wai-app-static`'s `staticApp` with two overrides:

1. **MIME type fix** — `ssGetMimeType`: files with no extension (all compiled wiki pages) get `text/html; charset=utf-8` instead of `application/octet-stream`
2. **Index file** — `ssIndices`: tries `index`, `index.html`, `index.htm` so `/` routes directly to the homepage

Verified working with curl: both `/` and `/index` return `Content-Type: text/html; charset=utf-8` and full HTML.

## Key Decisions

### Why a custom `serve` command instead of patching Hakyll's `watch`
Hakyll's built-in `watch`/`server` uses `wai-app-static` internally but doesn't expose MIME type configuration through `Configuration`. Patching hakyll would require a source override in `cabal.project`. Adding our own `serve` subcommand keeps all code local and is trivial to maintain.

### Why `--no-server` on `watch`
Running `cabal run wiki -- watch` starts Hakyll's broken server on port 8000 at the same time as auto-rebuilding. Use `--no-server` to suppress it so our `serve` command can own port 8000.

### `-threaded` GHC option (added earlier in session)
Without `-threaded`, Hakyll's `watch` command crashes immediately after building with `GHC.Event.Thread.getSystemTimerManager: the TimerManager requires linking against the threaded runtime`. The warp server and file watcher both require the threaded RTS. Fixed by adding `-threaded -rtsopts -with-rtsopts=-N` to `ghc-options` in `wiki.cabal` and doing a clean rebuild (`cabal clean && cabal build`).

### Why not `python3 -m http.server`
Python's default HTTP server serves extensionless files as `application/octet-stream`, causing the same download problem. A custom Python class with `extensions_map = {'': 'text/html'}` would work but requires a separate script. The Haskell `serve` command is preferable since it's invoked the same way as all other wiki commands.

## Local Dev Workflow

```bash
# Terminal 1 — auto-rebuild on file changes, no built-in server
source ~/.ghcup/env
cabal run wiki -- watch --no-server

# Terminal 2 — serve _site/ with correct MIME types
source ~/.ghcup/env
cabal run wiki -- serve
```

Then open `http://127.0.0.1:8000/` — serves the `index` file directly as HTML.

For one-shot (no auto-rebuild):
```bash
cabal run wiki -- build
cabal run wiki -- serve
```

## Context for Future Sessions

- The site pages have **no file extension** by design (cool URLs for production on S3/CloudFront). The `EXT` environment variable in `Main.hs` controls this; it defaults to `""`.
- Internal links between pages (e.g. `./About`, `./Links`) are extensionless and work correctly in production. Locally they work because `cabal run wiki -- serve` serves extensionless files as HTML.
- The `index` file at the repo root is **pre-built static HTML** (not a `.page` Markdown file). It's copied to `_site/index` verbatim by Hakyll's static file rules.
- `_site/` is gitignored. It is regenerated by `cabal run wiki -- build` or `watch`.
- All Haskell tooling requires `source ~/.ghcup/env` first (GHC 9.6.6, cabal 3.16.1.0).
